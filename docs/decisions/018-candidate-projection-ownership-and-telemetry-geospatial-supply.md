# ADR-018: Candidate-Projection Ownership and Telemetry Geospatial Supply

**Status:** Accepted
**Date:** 2026-06-30

## Context

Dispatch selects ride candidates by reading a `NearbyAvailableDrivers` view — driver location ⋈ availability ⋈ vehicle-class capability, filtered by pickup radius ([W001 §5.3](../workshops/001-dispatch-event-model.md)). The slice walk modeled that view *agnostically*: it assumed the view "is available" and explicitly deferred how it gets populated to [§10 parking-lot #4](../workshops/001-dispatch-event-model.md) and [§11 ADR-candidate #3](../workshops/001-dispatch-event-model.md). The consumer side is already settled — `CandidatesSelected` is a Klefter Translation-in decision-event capturing search parameters and policy version — but the *supplier* side has been open since 2026-04-24 and is the single dashed-line dependency on [context-map edge #5 (Telemetry → Dispatch)](../context-map/README.md#5-telemetry--dispatch-deferred).

The parked question is ownership-shaped: does the nearby-drivers view live **inside Telemetry/Driver Profile as a queryable index that Dispatch calls at selection time**, or **inside Dispatch as a local projection fed by upstream events**? The two shapes are both customer-supplier relationships and both hand Dispatch the data it needs, but they imply different consistency guarantees, different transports, and different boundary-ownership. The choice cannot be deferred past the Telemetry workshop (W006), because W006 must model how Telemetry *supplies* location — and that supply contract is meaningless until ownership is decided.

The view is a **join of two different flow shapes**, and [ADR-005](../decisions/005-transport-selection-by-flow-type.md) already classifies them oppositely. Driver *location* is high-volume, append-only, lag-tolerant telemetry — ADR-005 routes that to **Kafka**. Driver *availability* (`DriverCameOnline` / `WentOnBreak` / `WentOffline` / `VehicleChanged`) is low-volume, correctness-sensitive business state — ADR-005 routes that to **ASB**. Any ownership decision has to honor that split rather than collapse both halves onto one transport.

There is also a standing tension with the vision document, which describes Telemetry as the BC that "maintains the geospatial index *that Dispatch queries*" — phrasing that presupposes the queryable-index shape and the gRPC-query answer. This ADR confronts that phrasing directly rather than quietly contradicting it.

## Options Considered

### Option A — Telemetry-owned queryable geospatial index (Dispatch queries via gRPC)

Telemetry ingests GPS, maintains the authoritative geospatial index as its own queryable view, and exposes a `GetNearbyDrivers(pickup, radius, vehicleClass)` gRPC **unary** call. Dispatch queries it synchronously inside `CandidateSelectionAutomation` at selection time. This is the shape the vision doc's "index that Dispatch queries" wording assumes.

The appeal is conceptual tidiness: location data lives in exactly one place, owned by the BC whose job is location, and Dispatch holds no geo state of its own. It matches the naïve reading of "Telemetry owns location-of-record."

The objections are decisive. First, it puts a **synchronous cross-service call on Dispatch's hottest path** — every candidate-selection round blocks on a Telemetry RPC, coupling Dispatch's matching availability to Telemetry's. Hyperscale matchers do the opposite: they keep an in-matcher supply index precisely so the matching loop never round-trips for supply (`docs/research/ride-sharing-lessons-learned.md`). Second, and most consequential for a project whose *entire reason for existing* is to showcase the transports, it leaves **Kafka with a producer and no consumer**: Telemetry would publish GPS to Kafka for its own index, but nothing downstream would consume that stream until surge pricing arrives (post-MVP). The first Telemetry build would demonstrate Kafka ingest with no Kafka-fed read model — a hollow showcase. Third, it still does not escape the join: availability lives in Driver Profile, so either Telemetry must also consume Driver Profile's availability (pulling an unrelated BC's business state into the telemetry index — a boundary violation) or Dispatch must join the gRPC result against availability anyway, which reintroduces local geo-adjacent state and defeats the "Dispatch holds none" appeal.

### Option B — Dispatch-owned local projection, fed by Kafka (location) + ASB (availability)

`NearbyAvailableDrivers` is a **Dispatch-owned local projection**. Telemetry publishes driver **location** to **Kafka** (`DriverLocationUpdated`, on a throttled / cell-changed grain — never per-ping); Driver Profile publishes **availability** transitions to **ASB**. Dispatch folds both streams into one local view and reads it synchronously, in-process, during candidate selection — the `INearbyAvailableDriversSource` stub seam built at slice 5.3 becomes this projection without reshaping the handler.

This makes the view a **multi-transport projection**: Kafka (geo) + ASB (availability) folding into a single Dispatch-local read model. That is the cleanest possible embodiment of ADR-005's "transport chosen per flow shape" — the two halves of one view ride the two transports ADR-005 assigns to their respective flow shapes, and the join happens where the *consumer's* invariant lives.

The cost is that Dispatch holds a geospatial projection — state derived from another BC's data — and that the supply is eventually consistent rather than read-your-writes. Both costs are addressed in the Decision below; neither is fatal, because Dispatch's matching path is explicitly designed to tolerate candidate staleness at the acceptance layer.

## Decision

**Option B.** The `NearbyAvailableDrivers` view is a **Dispatch-owned local projection**, not a Telemetry-owned queryable index. This ADR locks the **Telemetry (Kafka, location) supplier half** and the ownership/consistency model; it states the Driver Profile (ASB, availability) half only as a forward-constraint, to be locked at Driver Profile's own workshop.

**Ownership and supply.**

- **Location → Kafka.** Telemetry publishes `DriverLocationUpdated` to Kafka on a **throttled / cell-changed grain** (cell-change or heartbeat interval), never per raw ping. Raw GPS is ingested separately (gRPC client-streaming, per ADR-005) and processed in flight; the Kafka topic carries the throttled breadcrumb, and Dispatch is its first real consumer. The topic is **partitioned by `driverId`**, so per-driver ordering is a transport property rather than an application concern.
- **Availability → ASB (forward-constraint).** Driver Profile publishes availability transitions (`DriverCameOnline`, `DriverWentOnBreak`, `DriverWentOffline`, `DriverVehicleChanged`, each carrying `vehicleClass` capability) over ASB per ADR-005 and [ADR-014](../decisions/014-asb-topic-naming-convention.md). **This ADR does not lock that contract** — locking an un-workshopped BC's publication shape would be a cross-BC over-reach of exactly the kind the context-map discipline exists to prevent. It is recorded here as the structural pair to the Telemetry half and as a forward-constraint on [context-map edge #6 (Driver Profile → Dispatch)](../context-map/README.md#6-intra-actor-topology-deferred).
- **The join is Dispatch's.** Dispatch folds both streams into one local projection. Location ⋈ availability happens inside Dispatch, against Dispatch's own candidate-selection invariant. Telemetry is **availability-agnostic** — it tracks location for any authenticated, actively-pinging driver and never consumes Driver Profile; "active" means *pinging*, not *available*.

**Consistency guarantee.** The projection is **eventually consistent, last-writer-wins per driver**, keyed on a server-stamped monotonic field (`serverReceivedAt`, not the client-controllable `deviceTimestamp`). There is no read-your-writes guarantee. This is acceptable because the model **already absorbs candidate staleness at the acceptance layer**: selection optimizes for *recall* from a possibly-stale local view, and acceptance enforces *correctness* (exactly-one assignment) via the aggregate-per-invariant `OfferAccepted` guard and `OfferRevoked` ([W001 §5.5](../workshops/001-dispatch-event-model.md), [ADR-012](../decisions/012-aggregate-per-invariant.md)). A driver who went off-shift a moment ago may briefly survive in the projection; Dispatch's availability filter and the acceptance guard catch it. A staleness-ceiling exclusion (evicting positions older than a bound) is **deferred to v2** — keep v1 honest, simple, and working before tuning staleness.

**Override recorded.** This explicitly overrides the vision document's "the geospatial index *that Dispatch queries*" phrasing. The vision doc presupposed the Option A shape; this ADR commits Option B. [W001 §10 parking-lot #4](../workshops/001-dispatch-event-model.md) had already reopened the question by deferring it rather than treating the vision-doc wording as settled. The vision doc is updated to match when W006 ships.

**Scope boundary — what this ADR deliberately does not decide.** Three architectural questions surfaced alongside this decision and are carried as their own [W001 §11](../workshops/001-dispatch-event-model.md)-style candidates, not folded in here (one decision per ADR):

1. **Kafka topic-naming convention.** The location topic name extends [ADR-014](../decisions/014-asb-topic-naming-convention.md)'s `<source-bc>.<event-name-kebab>` shape, but ADR-014 is currently ASB-scoped. Generalizing it to a transport-agnostic convention is its own ADR, triggered when the first Kafka topic lands in the W006 build.
2. **Stream-processing as a fourth modeling shape.** Telemetry processes pings in flight and is *not* event-sourced (a per-ping-per-driver event store would detonate); its location-of-record is a last-known-position document, and the Kafka topic is the breadcrumb history. Whether that warrants a shape-ADR (parallel to the W004/W005 shape candidates) is decided at Telemetry's first implementation slice.
3. **Windowed gRPC client-streaming ingest.** The windowed-vs-per-shift ingest convention is its own question and may fit better as a `docs/skills/` skill than an ADR; decided at the first gRPC ingest slice.

This ADR decides **ownership, the Telemetry supply transport, and the consistency model** — nothing more.

## Consequences

Dispatch owns a geospatial projection — state derived from Telemetry's and Driver Profile's data. This is a deliberate, named cost: it trades the conceptual tidiness of "location lives in exactly one BC" for **Dispatch autonomy on its critical matching path** (no synchronous cross-service call per selection round) and for a **real Kafka consumer in the first Telemetry build**. For a reference architecture whose headline feature is its transports, a Kafka stream with a live downstream read model is worth materially more than a Kafka stream feeding only its own producer's index.

The decision makes `NearbyAvailableDrivers` the project's **first multi-transport projection** and the cleanest teaching example of ADR-005: one view, two flow shapes, two transports, joined at the consumer's invariant. Someone studying CritterCab to learn transport selection sees Kafka and ASB folding into a single Dispatch read model, with the transport choice driven entirely by the flow shape of each half.

The Telemetry workshop (W006) is now unblocked and inherits a firm supplier-side contract to model against: a throttled/cell-changed `DriverLocationUpdated` published to Kafka, partitioned by `driverId`, deduplicated on `serverReceivedAt`, carrying the H3 cell id as published-language vocabulary (both BCs must agree on cell system and resolution because Dispatch buckets by the same cell id for radius queries — contract vocabulary, like [ADR-013](../decisions/013-shared-cross-bc-identifier.md)'s canonical id, not an implementation detail). Telemetry's throttle policy is itself governed by configuration-as-events ([ADR-011](../decisions/011-configuration-as-events-bootstrap.md)) via a `TelemetryPolicyConfigured` singleton stream — the one event-sourced stream in an otherwise stream-processing BC.

Driver Profile's workshop inherits a forward-constraint, not a lock: it must publish availability transitions over ASB in a shape Dispatch's projection can fold, but the exact contract is its to design. [Context-map edge #5 (Telemetry → Dispatch)](../context-map/README.md#5-telemetry--dispatch-deferred) becomes **solid on the Telemetry (supplier) side**; [edge #6 (Driver Profile → Dispatch)](../context-map/README.md#6-intra-actor-topology-deferred) is **informed but stays dashed** until that workshop runs.

The eventual-consistency choice means Dispatch can, in rare windows, select a driver whose true state has changed. This is not a defect introduced here — it is the same staleness the model already tolerates by design, now made explicit and bounded to the acceptance layer's reconciliation. If production load reveals pathological staleness, the v2 staleness-ceiling exclusion (and, if needed, the bidirectional-streaming ingest upgrade ADR-005 anticipates) reopen the tuning question without reshaping this decision.

This decision **resolves [W001 §11 ADR-candidate #3](../workshops/001-dispatch-event-model.md)** and **closes [W001 §10 parking-lot #4](../workshops/001-dispatch-event-model.md)**. The three scope-boundary candidates above remain open with their named triggers.
